Issue · Pull Request · Projects

GitHub の3つのコア機能をゼロから理解する

#01 Issue #02 Pull Request #03 Projects #04 連携フロー #05 まとめ
01

Issue

一言で：リポジトリに紐付いた「タスクチケット・メモ・バグ報告票」

Issue は GitHub 上のスレッド1件1件のこと。 バグ報告・機能要望・TODO・議論など、目的は何でもよい。 番号（#12 のような形式）が自動付与され、 PR やコミットから参照できる。

🐛
バグ報告
「〇〇したらクラッシュする」「エラーが出る」など再現手順と一緒に記録
✨
機能要望
「〇〇を追加したい」「UX を改善したい」など新機能のアイデアを残す
📋
TODO メモ
「リファクタリング」「ドキュメント更新」など後回しにしたいタスク
💬
議論の場所
「設計どうする？」「ライブラリどれ使う？」などチームでの意思決定ログ
Issue の構成要素
#12  「ログイン機能を実装する」          ← タイトル + 番号
 │
 ├─ 本文                              ← Markdown で書ける
 ├─ Assignee  @kichima              ← 担当者
 ├─ Label     enhancement           ← bug / enhancement / など
 ├─ Milestone v1.0                  ← 期限単位のグループ
 ├─ コメントスレッド                   ← 議論をここに集約
 └─ 紐付いた PR  #34               ← マージ時に自動クローズ
💡
PR で自動クローズ
PR の説明文に Closes #12 と書くと、 その PR がマージされた瞬間に Issue #12 が自動でクローズされる。 手動で閉じる必要がなくなり、実装と課題が紐付いた状態で履歴に残る。
👤
個人開発でも Issue は有効
チーム用のツールという印象があるが、1人でも「なぜこの実装をしたか」の 意思決定ログとして機能する。後で読み返したとき文脈がわかる。
02

Pull Request

一言で：「このブランチを main にマージしていいか確認・レビューする仕組み」

機能開発は feature/xxx などのブランチで行い、 完成したら PR を作成して main へのマージを依頼する。 差分の確認・コメント・CI テストが1画面に集約されるのが特徴。

PR の基本フロー
main ブランチ
  └─ feature/login ブランチを切る
       │
       ├─ コミット・コミット・コミット
       │
       └─ git push origin feature/login
            │
            └─ GitHub で PR を作成
                 │  タイトル・説明・Closes #12 を記載
                 │
                 ├─ レビュアーがコメント → 修正 → 再レビュー
                 ├─ CI（GitHub Actions）が自動テスト
                 │
                 └─ 承認 → main へマージ → Issue 自動クローズ
🔍
差分の可視化
どのファイルの何行が変わったか一目でわかる。行単位でコメントを付けられる
⚙️
CI との連携
テストが失敗したらマージをブロックするルールを設定できる
🔀
マージ戦略
Squash merge / Rebase merge / Merge commit の3種類から選択できる
マージ戦略 コミット履歴 使いどころ
Squash merge 1コミットにまとめて main へ 作業中の細かいコミットを整理したいとき
Rebase merge コミットを並べ直して main へ 履歴を一直線に保ちたいとき
Merge commit マージコミットを生成 ブランチの存在を履歴に残したいとき
💡
個人開発でも PR 運用は有益
直接 main にコミットせず、PR 経由でマージする習慣を作るだけで main が常にクリーンな状態に保たれる。 「なぜこの変更をしたか」の説明も PR の本文に残せる。
03

Projects

一言で：Issue・PR をカードとして並べる「タスク管理ボード」（Trello / Jira に近い）

リポジトリの Issue / PR を自動で取り込み、カンバン・テーブル・ロードマップなど 複数の形式で進捗を管理できる。
現在は Projects（新） が推奨。 旧 Classic Projects は非推奨になっている。

📌
ボード表示
Todo / In Progress / Done のカンバン形式でカードを管理
📊
テーブル表示
スプレッドシートのように Issue を一覧化してソート・フィルタ
🗺️
ロードマップ
時系列でタスクの開始・終了を Gantt チャート的に可視化
⚡
自動化
PR マージ → Done に移動、Issue クローズ → アーカイブ、など
Projects のデータ構造イメージ
Project: Kage v1.0
  │
  ├─ 【Todo】
  │    ├─ #5  transit delay アラート実装
  │    └─ #8  通知チャンネル設定 UI
  │
  ├─ 【In Progress】
  │    └─ #3  行動パターン学習ロジック  ←  PR #21 紐付き
  │
  └─ 【Done】
       └─ #1  初期セットアップ  ← Closed
📝
カスタムフィールドが便利
優先度（High / Medium / Low）・担当者・期日・ストーリーポイントなど 独自フィールドを自由に追加できる。スプレッドシートで管理していたものを そのまま GitHub に持ち込めるイメージ。
04

3機能の連携フロー

3つを組み合わせると、開発の流れが Issue 起点で一本化される。

Projects にタスク追加
→
Issue 起票 #12
→
ブランチ切る
→
実装・コミット
→
PR 作成
Closes #12
→
レビュー・CI
→
マージ
→
Issue 自動クローズ
→
Projects → Done
🔗
全部使わなくてもいい
最初は Issue だけでも十分。慣れてきたら PR 運用を追加し、 タスクが増えてきたら Projects を導入する、という段階的な取り入れ方が現実的。
